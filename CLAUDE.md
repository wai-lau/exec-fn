# exec-fn

Personal automation system designed to support executive function (ADHD scaffolding).

## Architecture

- **Droplet**: DigitalOcean NYC1, reserved IP 168.144.13.51, domain wai-lau.net
- **nginx**: bare-metal, handles SSL termination, proxies to Docker on port 8080
- **FastAPI**: serves static files and all pipeline API endpoints
- **Docker**: single container locally and on Droplet, same compose file
- **Cron**: runs inside the container, triggers `POST /api/morning` at 4:30 AM ET daily

## Stack

- Python / FastAPI
- Docker / docker-compose
- nginx (bare-metal on Droplet only)
- Anthropic API (claude-sonnet-4-6) for directives, delta, omens
- Google Calendar API for omens
- rmapi + rmscene for reMarkable integration

## Terminology

| Term | Meaning |
|------|---------|
| **directives** | Today's actionable tasks, generated by Claude. 3 easy / 3 medium / 1 hard. |
| **omens** | Upcoming reminders â€” things that didn't fit in directives. One-liners (title + date), easy to reference when asking Claude to reprioritize. Sourced from Google Calendar. |
| **r&d** | Future projects with no fixed deadline â€” ongoing creative / technical work. |
| **delta** | Claude's interpretation of what Wai actually did today, based on handwritten feedback on the reMarkable. Used to adjust tomorrow's directives. |
| **archive** | All past WAI PDFs â€” primarily for debugging and backup. |

## Morning pipeline (`POST /api/morning`)

1. `build_pdf.build(WAI_<ts>.pdf)` â€” builds timestamped PDF from `directives.json` sections + `rd.json`
2. `rmapi put --force WAI_<ts>.pdf` â€” uploads to reMarkable
3. `pull_exec(baseline=True)` â€” pulls EXEC.rmdoc, saves as `EXEC_<ts>.rmdoc` and copies to `EXEC_<date>_baseline.rmdoc`

Triggered automatically at **4:30 AM ET** by cron inside the container.

## Delta pipeline (`POST /api/delta`)

1. Pull latest EXEC.rmdoc (timestamped)
2. Find today's baseline (`EXEC_<date>_baseline.rmdoc`) â€” created by morning build
3. Render latest page 0 to PNG via `rm_to_pdf.rmdoc_to_image()`
4. Send image to Claude API: "describe Wai's handwritten feedback, suggest adjustments to tomorrow's directives"
5. Save to `delta.json`: `{analyzed_at, baseline_file, latest_file, wai_notes, adjustments}`

## Directives pipeline (`POST /api/directives`)

Reads `rd.json` + `omens.json` + `delta.json` + `preferences.json`, sends to Claude. Optional `feedback` string in POST body is appended to the prompt.

Saves to `directives.json`:
```json
{
  "generated_at": "...",
  "easy": ["task 1", "task 2", "task 3"],
  "medium": [
    {"title": "task", "steps": ["step 1", "step 2"]},
    {"title": "task", "steps": ["step 1", "step 2"]},
    {"title": "task", "steps": ["step 1", "step 2"]}
  ],
  "hard": {"title": "task", "steps": ["step 1", "step 2", "step 3"]}
}
```

3 easy (5â€“15 min each), 3 medium (30â€“60 min, broken into sub-steps), 1 hard (1â€“2 hours, broken into sub-steps). Word count roughly equal across columns.

## Omens pipeline (`POST /api/omens`)

Fetches next 14 days of Google Calendar events, sends to Claude to flag anything needing prep, saves to `omens.json`.

### Google Calendar setup (one-time)

1. Create Google Cloud project, enable Calendar API, download `credentials.json` (Desktop app type)
2. Copy into container: `docker compose cp credentials.json api:/root/.config/gcal/credentials.json`
3. SSH tunnel: `ssh -L 8765:localhost:8765 root@wai-lau.net`
4. Run auth: `docker compose exec -it api python3 gcal_auth.py`
5. Visit the printed URL, paste the code back
6. Token saved to `gcal-auth` Docker volume at `/root/.config/gcal/token.json` (auto-refreshes)

## reMarkable Integration

### Tools
- **ddvk/rmapi v0.0.32** â€” the only working CLI tool as of Apr 2026. `juruen/rmapi` is dead (410 Gone). Installed in Docker container via tar.gz from GitHub releases.
- rmapi talks to reMarkable cloud over HTTPS â€” works anywhere.
- Auth stored in Docker named volume `rmapi-auth` at `/root/.config/rmapi`. Re-auth required on Droplet separately.
- No locking â€” "last write wins". Don't upload while the notebook is open on the device.

### Uploading back to reMarkable
- `rmapi put <file>.rmdoc` â€” creates new document
- `rmapi put --force <file>.rmdoc` â€” replaces existing document with same name
- Cannot update an existing document by UUID â€” must use a new UUID and new `visibleName`, or `--force` on same name
- Set `metadata["new"] = True` when packaging a new rmdoc for upload

### .rmdoc format
- ZIP archive containing: `{uid}.content`, `{uid}.metadata`, `{uid}/{page_id}.rm`
- `.rm` files are reMarkable v6 binary format, parsed/written with `rmscene`
- `write_blocks(file_obj, blocks)` â€” note arg order (file first)

### rmscene imports
- `ParagraphStyle`, `Text` (the class used as RootTextBlock.value) â†’ `from rmscene.scene_items import ParagraphStyle, Text`
- `TextDocument`, `Paragraph`, `CrdtStr` are a newer higher-level API in `rmscene.text` â€” not used for raw block construction
- Do NOT import from `rmscene.text` for block writing â€” use `scene_items` and `crdt_sequence` directly

### rmscene (Python)
- Use `git+https://github.com/ricklupton/rmscene.git` (v0.8.1.dev0) â€” PyPI 0.8.0 has same limitations
- "Some data has not been read" warning = PathItemBlock (ITEM_TYPE 0x04) not yet supported â€” newer stroke format
- `read_blocks(f)` â†’ list of blocks
- `RootTextBlock` â€” contains typed text. Has `.value` (Text object) with `.items` (CrdtSequence), `.styles`, `.pos_x`, `.pos_y`, `.width`
- `CrdtSequence._items` â€” internal dict, directly mutable for filtering/rebuilding
- `CrdtSequence.values()` â€” yields text values in order (ints 1/2 for heading markers, strings for text)
- `CrdtSequence.sequence_items()` â€” yields `CrdtSequenceItem` objects in order
- `SceneLineItemBlock` â€” handwritten strokes. `.item.value.points` = list of Points with `.x`, `.y`
- `TreeNodeBlock` â€” layer definitions. `.group.label.value` = layer name (e.g. "Ink", "AI", "Wai")

### ParagraphStyle enum
- BASIC=0, PLAIN=1, HEADING=2, BOLD=3, BULLET=4, BULLET2=5, CHECKBOX=6, CHECKBOX_CHECKED=7
- Styles dict maps `predecessor_CrdtId â†’ LwwValue(timestamp, ParagraphStyle)`
- `CrdtId(0,0)` as key = style for the first paragraph

### Coordinate system
- reMarkable page: 1404 Ă— 1872 px
- **Stroke coords**: x=0 is LEFT edge (0â€“1404) in some notebooks, x=0 is PAGE CENTER (-702 to +702) in others (e.g. EXEC template). Check stroke x range to determine.
- **RootTextBlock pos_x**: offset from page center. `pos_x=-468, width=936` = full page width
- **RootTextBlock pos_y**: from top of page (absolute)
- Stroke y coords in typed notebooks are relative to the text block's `pos_y` (not absolute page coords)
- `verticalScroll` in cPages.pages is NOT needed for stroke positioning â€” strokes relative to text origin

### PDF display scaling on reMarkable
- reMarkable displays PDFs **fit-to-height** (not fit-to-width). The PDF is scaled so its height = 1872px exactly, then centered horizontally.
- For A5 PDF (419.53 Ă— 595.28 pt): scale = 1872/595.28 = 3.145, rendered width = 1319px, left/right margin = 42px each.
- Stroke coordinates are in device screen space (1404Ă—1872). To render strokes over a PDF background:
  1. Scale PDF with `scale = min(RM_W/pg.rect.width, RM_H/pg.rect.height)` (height-fit)
  2. Center PDF horizontally: `x_start = (RM_W - img.width) // 2`
  3. Paste PDF at `(x_start, 0)` on a 1404Ă—1872 white canvas
  4. Draw strokes at `(x + x_offset, y)` where `x_offset = 702` for center-origin notebooks
- **Verified empirically**: height-fit aligns stroke clusters to within ~6px of PDF content positions.

### rmscene firmware compatibility
- Newer reMarkable firmware uses non-zero `block_id` in `RootTextBlock`. The assertion `assert block_id == CrdtId(0, 0)` in `rmscene/scene_stream.py` will crash on these files.
- Fix: patch via sed in Dockerfile after pip install (already applied).
- After the fix, blocks that were `UnreadableBlock` become parseable `RootTextBlock` objects.
- "Unrecognised text format code 8" warning = newer firmware data format, not fixable, reading still works.

### Writing text to reMarkable via rmscene â€” DOES NOT WORK RELIABLY
- Writing custom `RootTextBlock` via rmscene produces truncated text and wrong paragraph styles.
- Root cause: unclear, likely CRDT merge behavior incompatible with rmscene's output.
- **Do not attempt this approach.** Use PDF instead.

### PDF approach (working solution)
- Generate an A5 PDF using `reportlab`, upload with `rmapi put --force WAI_<ts>.pdf`.
- reMarkable renders PDFs natively and perfectly.
- `build_pdf.build(out_path)` generates a 2-page A5 PDF:
  - Page 1: daily tasks from `directives.json` items + reminders
  - Page 2: r&d projects in 2-column layout from `rd.json`
- Run manually: `docker compose exec api python3 -c "from build_pdf import build; build()"`
- Or trigger via API: `POST /api/morning`

### EXEC notebook structure (as of Apr 2026)
- Page 0: 4 layers: Ink, AI, Wai (plus unnamed base). Left column = active quests + reminders. Center-origin coords.
- Page 1: 3 layers: AI, Wai (plus unnamed base). R&D content page.
- AI layer approach abandoned â€” rmscene text writing unreliable (see above).
- **WAI.pdf** is the primary AI-generated document. Uploaded separately via `rmapi put --force`.
- Wai writes feedback as handwritten strokes on the Wai layer â€” read by delta pipeline.

### Rendering .rmdoc to PNG
- `rm_to_pdf.rmdoc_to_image(path, page_index)` â€” renders strokes + text to PNG bytes using Pillow
- For PDF-based rmdocs: renders the embedded PDF as background (height-fit, centered), then overlays strokes
- For native notebooks: renders text blocks from rmscene, then overlays strokes
- Center-origin stroke coords (min x < -10) use `x_offset = 702`; left-origin use `x_offset = 0`
- Output PNG is half-resolution (702Ă—936 or proportional) for Claude vision API
- Used by delta pipeline to send page images to Claude for vision analysis
